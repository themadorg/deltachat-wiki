# وضعیت مشترک برنامه وب

در یک برنامه وب چند کاربره معمولی،
کلاینت‌ها سورس برنامه را از یک سرور HTTP مرکزی دانلود می‌کنند
و برای ارسال پیام به دیگران به آن تکیه می‌کنند.
در این [مدل کلاینت-سرور](https://en.wikipedia.org/wiki/Client%E2%80%93server_model)،
سرور می‌تواند از موقعیت مرکزی و مرجع خود
برای اطمینان از اینکه تمام کلاینت‌های متصل پیام‌ها را به همان ترتیب دریافت می‌کنند، استفاده کند.

اگرچه _برنامه‌های webxdc_ می‌توانند در شرایطی استفاده شوند
که همه اعضای یک چت مشخص توسط یک سرور واحد میزبانی می‌شوند،
مشخصات فرض نمی‌کند که این مورد خواهد بود
و در واقع برای پشتیبانی از هر دو حالت فدراسیونی (چند سروری)
و کاملاً [همتا به همتا](https://en.wikipedia.org/wiki/Peer-to-peer) طراحی شده است.
علاوه بر این، پیام‌رسان‌های پشتیبان webxdc که پیام‌رسانی آفلاین-اول را پیاده‌سازی می‌کنند
تأخیرهای دلخواه در تحویل پیام ایجاد می‌کنند.
کلاینت‌های مختلف ممکن است به‌روزرسانی‌ها را با ترتیب‌های بسیار متفاوتی دریافت کنند،
و در نتیجه آن به‌روزرسانی‌ها را در دنباله‌هایی اعمال کنند که نتایج متناقض تولید می‌کنند.

استراتژی‌ها و فناوری‌های متنوعی
برای اجتناب یا حل چنین تناقضاتی وجود دارد.
مشخصات webxdc عمداً درباره استفاده آن‌ها بی‌طرف است،
و به نویسندگان برنامه اجازه می‌دهد رویکردی را انتخاب کنند
که برای نیازهایشان مناسب‌تر است.

این فصل:

* روش‌های رایجی که وضعیت می‌تواند در زمینه همتا به همتا ناهماهنگ شود را توصیف می‌کند
* مقدمه‌ای بر **CRDTها** (نوع‌های داده تکرارشونده بدون تناقض) ارائه می‌دهد
و شرایطی که استفاده از آن‌ها مفید است را نشان می‌دهد
* نحوه تطبیق برنامه برای استفاده از CRDTها
برای همگام‌سازی مداوم وضعیت مشترک
بین چندین کاربر یا دستگاه با استفاده از مثال‌های عملی را نشان می‌دهد
